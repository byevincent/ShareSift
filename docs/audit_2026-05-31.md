## v0.5 metrics audit — 2026-05-31

Second audit pass on ShareSift, following the v0.5 Phase B/C ship (Linux
corpus, two-model split, pysnaffler hook). Where the
[2026-05-30 audit](audit_2026-05-30.md) caught **data leakage** before
shipping headline numbers, this pass interrogates the **metrics
themselves**: are the calibration claims real, do the recall numbers
come with confidence intervals, do the cross-source generalization
claims survive a held-out test, is the labeled corpus actually clean.

Six Tier-1 measurement audits surfaced three honest defects and three
defensible wins. One Tier-2 model fix (Windows OOD recalibration) was
landed in the same pass; two more (verified-positive content rebuild,
field-comparable benchmarks) are deferred to v0.6 with documented scope.

## Context

v0.5 shipped on 2026-05-31 with the following claims in the README:

| Metric | v0.5 (reported) |
|---|---|
| Path classifier — Windows PR-AUC on Snaffler-blind benchmark | 0.985 |
| Path classifier — Windows recall vs Snaffler at threshold 0.5 | +13.7 pp |
| Path classifier — Linux PR-AUC on Linux-rule-blind benchmark | 0.99 |
| Content classifier F1 vs Wiz baseline | 0.971 vs ~0.838 |

The audit asks: of these four headline numbers, which survive a
methodologically careful re-measurement, which need footnotes, and
which are wrong?

## What was checked

Four parallel research agents independently combed v0.5's training,
calibration, evaluation, and comparable published work for measurement
gaps. They converged on six high-leverage low-effort audits (the
Tier-1 sprint) plus three deeper fixes (the Tier-2 sprint). All
Tier-1 + the first Tier-2 ran here; the remaining two Tier-2 items
are scoped and queued.

**Tier 1 — measurement audits:**

* **1.A Calibration.** Expected Calibration Error + 10-bin reliability
  tables + Brier + MCC@0.5 per model × eval set
  (`tools/calibration_audit.py`).
* **1.B Paired bootstrap CI on the Snaffler-beat claim.** 10,000-resample
  bootstrap on Windows recall@0.5 vs the Snaffler-rule recall baseline
  (`tools/snaffler_comparison_ci.py`).
* **1.C Label-noise estimation.** 5-fold CV + Cleanlab confident
  learning on both Windows and Linux training corpora
  (`tools/label_noise_audit.py`).
* **1.D Per-secret-type content-classifier audit.** Regex-based type
  inference + per-type P/R/F1 over the 481-record content test split
  (`tools/per_secret_type_audit.py`).
* **1.E Cross-source held-out eval.** Train on one source, test on
  another (`tools/cross_source_eval.py`).
* **1.F Linux Snaffler-equivalent rule baseline.** Hand-curated 38-rule
  pack + rule-blind benchmark + ML-vs-rules comparison
  (`tools/linux_rules_baseline.py`).

**Tier 2 — model fixes:**

* **2.1 Windows OOD recalibration.** Refit the Windows calibrator on
  (train ∪ 30% of Snaffler-blind benchmark) so it sees the OOD
  distribution (`tools/recalibrate_windows_ood.py` audit, then
  `tools/install_recalibrated_windows.py` deploy).
* **2.2 Content corpus rebuild with verified positives.** Replace the
  ~96%-regex-only-labeled positives with Kingfisher-validated
  positives. **Deferred — multi-day scope. Tier-2.3 result below
  sharpens the case for this fix.**
* **2.3 Field-comparable CredData benchmark.** Direct comparison vs
  Biringa & Kul 2025 (Mistral-7B-Instruct-v0.3 LoRA, F1=0.985 on full
  CredData). Built `tools/build_creddata_benchmark.py`, downloaded
  Samsung's CredData via their `download_data.py` (333 repos, 1.0 GB
  obfuscated source), ran v0p3 over a 1500-record balanced stratified
  sample (500 positive / 1000 negative, snippets capped at 4000 chars
  for KV-cache budget). **Major finding: v0p3 F1=0.564 on CredData vs
  F1=0.971 on ShareSift's own test split — a 40.7 pp drop.** Precision
  collapses (0.984 → 0.445) while recall stays decent (0.958 → 0.768).
  Discussed in §Tier-2.3 finding below.

## Tier-1 findings

### 1.A — Calibration

| Severity | Finding | Numbers |
|---|---|---|
| ok | Windows model is excellently calibrated in-distribution. | ECE=0.007, Brier=0.003 on the 2150-record test split. |
| ok | Linux model calibration is good across distributions. | ECE=0.04 on test split, similar on Linux-rule-blind benchmark. |
| **error** | **Windows model is severely miscalibrated OOD.** | **ECE=0.30 on the 500-record Snaffler-blind benchmark, Brier=0.23.** The reliability table shows records the model predicts at 0.14 are actually 96% positive — the ranking is right but absolute probabilities don't transfer. |
| **error** | **Black tier band ≥0.95 captured ZERO Snaffler-blind records.** | The tier contract ("Black ≈ 95-100% positive rate") is meaningless OOD because the calibrated probabilities collapse toward zero on out-of-distribution paths. |

Mechanism: the v0.3 CalibratedClassifierCV was trained on
in-distribution data only. LightGBM's raw scores rank OOD records
correctly (PR-AUC 0.985) but the isotonic calibrator learned a mapping
that's only valid where the training-distribution's score density
matches. The fix is in §Tier 2.1.

### 1.B — Paired bootstrap CI on the Snaffler-beat claim

| Severity | Finding | Numbers |
|---|---|---|
| **error** | **The "+13.7 pp vs Snaffler" claim was measured on the raw uncalibrated model.** | Raw LightGBM at τ=0.5: recall=0.568. README cited this figure. |
| **error** | **The deployed calibrated model's recall@0.5 was −2.3 pp vs Snaffler.** | Calibrated point recall=0.392, CI [0.331–0.452]. Snaffler baseline=0.415. Delta=−0.023, CI [−0.084 to +0.037] — straddles zero. |
| ok | PR-AUC (calibration-invariant) is unaffected. | 0.985 on Snaffler-blind. Ranking claim holds. |

Mechanism: ECE-0.30 calibrator pushed nearly all predictions below
0.5 even when the underlying ranking was correct. The threshold-0.5
operating point — the natural comparison against a binary
Snaffler-flagged/unflagged label — sat in a calibration dead zone.
Fix in §Tier 2.1.

### 1.C — Label-noise estimation (Cleanlab)

| Severity | Finding | Numbers |
|---|---|---|
| ok | Windows labels are clean. | 86/11190 (0.8%) flagged. Noise matrix diagonal [0.997, 0.899] — ~10% of juicy-class labels potentially noisy. |
| info | Linux labels are noisier than Windows. | 109/1685 (6.5%) flagged. Noise matrix diagonal [0.863, 0.967] — 13.7% of not_juicy-class noisy. |
| info | Top Windows suspects are placeholder-y paths. | `\\path_to_store_pass\passwords.csv`, `\\MACHINE1\HKLM\SYSTEM` (registry-path syntax) — legitimate to flag for review. |
| info | Top Linux suspects sit in known labeling-tension zones. | `~/.ssh/host_key.rsa.pub` (correctly not_juicy per recent .pub-suffix calibration but trained without the signal at folding time), `/etc/nginx/certs/dhparam.pem` (public material — defensible either way). |

Confirms expectation: Windows had multiple Codex audit cycles + Vincent
sign-off; Linux had a single LLM-pass. Linux is the natural target if a
v0.6 label-cleanup pass happens, but the noise rate is low enough that
it's not blocking.

### 1.D — Per-secret-type content classifier

| Severity | Finding | Numbers |
|---|---|---|
| ok | Performance is roughly uniform across secret types. | hardcoded_password_assignment, db_connection_string, ssh/tls_private_key_pem all at P=R=F1=1.0 (small samples). generic_credential_mention (107 records, 102 positives) at F1=0.975. "other" catch-all (333 records) at F1=0.938. |
| info | "Other"-category FNs are mostly test code mentioning credential keywords without containing secrets. | 5 false negatives, all in the kingfisher-unverified-positive category — closer to label noise than model error. |

No hidden hotspot of the form "model excellent overall but blind to
SSH keys". Closes the "weak class hiding under the headline" hypothesis.

### 1.E — Cross-source held-out eval

| Severity | Finding | Numbers |
|---|---|---|
| **warn** | **Windows model's 0.985 PR-AUC is meaningfully within-source-inflated.** | Train Stack Exchange → test GitHub-Code-Search: PR-AUC=0.760, F1@0.5=0.85. 21 pp drop. |
| info | Linux model generalizes well across sources. | Train GitHub → test seed-mined: PR-AUC=0.934, F1=0.88. Only −4 pp from headline. |
| info | Reverse-direction Windows is unreliable due to small training set. | GH-only train: 657 records (too few). PR-AUC=0.31 reflects training-size limit, not real generalization gap. |

The Windows headline is real on the v0.5 measurement methodology but
the cross-source drop is large enough to footnote. Linux side is
honest end-to-end on this axis.

### 1.F — Linux Snaffler-equivalent rule baseline

| Severity | Finding | Numbers |
|---|---|---|
| ok | Hand-curated 38-rule Linux pack: precision-pure but low recall. | On 1685-record Linux corpus: P=1.000, R=0.294, F1=0.454. |
| ok | Linux ML model beats the rule pack on rule-blind paths. | On `linux_rule_blind_benchmark.jsonl` (500 records stratified by label, all rule-pack-silent): ML PR-AUC=0.99, F1@0.5=0.964 → **+52 pp F1, +96.8 pp recall** over rules. |

The proper apples-to-apples Snaffler-equivalent comparison for the
Linux side, which the v0.3 audit had left missing.

## Tier-2 fix that landed: 2.1 Windows OOD recalibration

The Tier-1.A and 1.B findings are the same underlying defect:
in-distribution-only calibration. Four recalibration strategies were
evaluated on a 30% / 70% split of the Snaffler-blind benchmark
(calibration / held-out):

| Variant | In-distribution ECE | OOD ECE | OOD Brier |
|---|---|---|---|
| baseline (current calibrated) | 0.007 | 0.298 | 0.225 |
| baseline (raw uncalibrated) | 0.003 | 0.227 | 0.196 |
| **isotonic on (train ∪ 30% bench)** | **0.006** | **0.192** | **0.151** |
| platt on (train ∪ 30% bench) | 0.006 | 0.239 | 0.213 |
| temperature on OOD only (T=5.52) | 0.121 | 0.225 | 0.155 |
| temperature on (train ∪ 30% bench) (T=1.66) | 0.005 | 0.233 | 0.185 |

The isotonic-mixed variant is the only strategy that improves OOD
calibration **without** regressing in-distribution. Temperature-mixed
matches in-distribution but doesn't move OOD enough; platt-mixed
underperforms isotonic on the held-out portion.

**Deployed (`models/path_classifier_v0_windows/calibrated.joblib`,
backup at `calibrated_v0p3.joblib`):**

| Metric | Before 2.1 | After 2.1 | Δ |
|---|---|---|---|
| In-distribution ECE (test split) | 0.007 | 0.006 | strict win |
| OOD ECE (Snaffler-blind held-out) | 0.298 | 0.192 | **−35%** |
| OOD MCC@0.5 | 0.49 | 0.73 | **+0.24** |
| OOD Bootstrap recall@0.5 | 0.392 [0.331–0.452] | **0.708 [0.650–0.763]** | **+31.6 pp** |
| OOD Delta vs Snaffler @0.5 | −0.023 [−0.084 to +0.037] | **+0.293 [+0.235 to +0.348]** | **CI no longer straddles zero** |
| Canonical OOD target: `GPP.xml` | None (0.28) | Yellow (0.667) | tier recovery |
| Canonical OOD target: `sql_backup.bak` | Yellow | Black | tier promotion |

Mechanism: `_OODCalibratedModel` (added to `src/sharesift/path.py`,
lines 43-73) wraps the raw LightGBM model + a single
IsotonicRegression fit on combined train + 30% benchmark scores.
Both halves of the joblib pickle live in importable module locations,
so runtime load succeeds without the deploy script in the import path.

Known limitation: sklearn-onnx has no registered converter for the
custom wrapper, so the ONNX export from v0.3 is stale. Python-runtime
deployment is unaffected. Custom-converter work is queued for v0.6.

## Tier-2.3 finding — CredData benchmark exposes content-classifier overfit to LLM-rule-labeled positives

The first field-comparable benchmark run reveals the gap between
"performs well on internal test split" and "performs well on an
externally-curated credential corpus" is large:

| | ShareSift v0p3 own test split | ShareSift v0p3 on CredData (1500 records) | Biringa & Kul 2025 on full CredData |
|---|---|---|---|
| Precision | 0.984 | **0.445** | (not separately reported) |
| Recall | 0.958 | **0.768** | (not separately reported) |
| F1 | **0.971** | **0.564** | **0.985** |
| Accuracy | 0.977 | 0.604 | (not separately reported) |

Predictions on CredData skew strongly positive: 862 "yes" / 638 "no"
on a corpus with 500 true positives / 1000 true negatives — the model
predicts twice as many positives as actually exist. False-positive
rate is 47.8% (478 FP / 1000 negatives); false-negative rate is 23.2%
(116 FN / 500 positives).

Mechanism, per the Tier-1.D per-secret-type finding: ~96% of the
content classifier's training positives are LLM-rule-labeled snippets
flagged by regex patterns for `password=`, `apikey=`, etc., not
Kingfisher-validated against live APIs. The model learned "this
snippet looks like it mentions credentials" rather than "this snippet
contains a credential value". On ShareSift's own test split — drawn
from the same LLM-rule labeling distribution — that distinction
doesn't matter and the metrics are strong. On CredData, where
negatives are hand-labeled test fixtures, parameter names, and
documentation that *deliberately* look credential-shaped without
containing secrets, the conflation collapses precision.

Caveats on the comparison:

* **Subset, not full corpus.** ShareSift ran on a 1500-record stratified
  sample (CredData has 66,896 labeled rows across 333 repos). The
  natural CredData ratio is ~3.2:1 negative-to-positive; the sample is
  2:1 to keep eval cost bounded. A full-corpus run might shift F1 by
  a few points either direction.
* **Snippets capped at 4000 chars.** ~4.1% of CredData snippets exceed
  4000 chars (worst case: a 173kB minified single-line file); these
  get truncated around the secret-bearing line. Biringa & Kul use a
  similar windowing approach. Marginal impact on the metric.
* **Model size asymmetry.** Biringa & Kul use Mistral-7B-Instruct-v0.3
  (4× more parameters than Qwen3-1.7B). The Wiz baseline is also a 7B
  model. Some of the gap is the model-size choice; some is the
  labeling-quality issue. The labeling-quality issue is the
  actionable one.
* **Training-distribution mismatch.** ShareSift trained on whole-file
  snippets from a kingfisher sweep; CredData is line-level with ±5
  context lines. Different snippet shape than the model saw at
  training time. This is partly the labeling-quality issue (the rule-
  labeled corpus skews toward whole-file patterns) and partly a
  framing choice (ShareSift's deployment use case is whole-file
  scoring, not line-level).

What this *is*: the strongest evidence so far that the Tier-2.2
verified-positive rebuild is the right next investment, with a
quantified-baseline number (F1=0.564) to beat. What this *is not*: a
refutation of the Wiz comparison on ShareSift's own test split — the
F1=0.971 there is real *for* that distribution, just not portable.

## Tier-2.2 partial finding — existing positives are not live-verifiable

Ran Kingfisher's full scan + live-validation pipeline
(`tools/validate_content_positives.py`) over the 1044 positive
snippets in `data/content/training_dataset.jsonl` (167 findings
raised, validations attempted against provider APIs):

| Status | Count | Percent | Meaning |
|---|---|---|---|
| Active | **0** | **0.0%** | Credential lives now + Kingfisher hit the provider API successfully |
| Matched but inactive | 86 | 8.2% | Kingfisher recognised the pattern; API rejected the credential (revoked, fake, never-real) |
| Matched but unknown | 43 | 4.1% | Kingfisher recognised the pattern; validation could not reach the API (DNS, firewall, target down) |
| Not matched | 915 | 87.6% | No Kingfisher rule fired at all — these are pure LLM-rule-labels |

The 0% live-active number is a stronger finding than Tier 1.D
estimated. Tier 1.D predicted "~38 kingfisher-VERIFIED" based on the
original training-time provenance metadata; that number was true *when
the corpus was assembled*. The credentials have since been revoked,
the synthetic ones never were live, and the unreachable-target slice
can't be re-validated without lifting egress restrictions. The 87.6%
pure-LLM-rule slice confirms the mechanism: most "positives" are not
credential strings the secrets-detection community would recognise —
they're keyword mentions in code that happen to match `password=` /
`apikey=` etc.

Implication for the v0.6 rebuild: **filtering the existing corpus
isn't viable**. A genuine "verified-positive" rebuild requires
starting from a fresh Kingfisher sweep over a fresh public-source
corpus, accepting only the records where Kingfisher's live validation
returns Active *at corpus-assembly time*, before the inevitable decay.
Multi-day scope and inherently time-limited (positives keep decaying).
Target: close the CredData F1 gap (0.564 → ≥0.85). The audit JSONL
(`data/content/positive_validation_audit.jsonl`) records the per-
positive verdict so a future plan can quantify exactly what would be
left if any partial reuse of the existing corpus is attempted.

  (Kingfisher validation must run synchronously with corpus
  assembly).

  Full single-file ONNX subgraph deferred — would need a custom

  numerically equivalent in the export verification.

## What the headline-numbers shift looks like

| | v0.5 (shipped) | v0.5 (audited & fixed) |
|---|---|---|
| Windows PR-AUC on Snaffler-blind | 0.985 | **0.985** (unchanged — PR-AUC is calibration-invariant) |
| Windows recall@0.5 on Snaffler-blind | 0.55 (raw model — misreported) | **0.71 [0.65–0.76]** (deployed calibrated) |
| Δ recall vs Snaffler @0.5 | +13.7 pp (raw — misreported) | **+29.3 pp [+23.5 to +34.8]** (deployed, with CI, significant) |
| Windows OOD ECE | 0.30 (not measured pre-audit) | **0.19** (after Tier 2.1) |
| Windows in-distribution ECE | 0.007 (not measured pre-audit) | 0.006 (after Tier 2.1 — strict win) |
| Linux PR-AUC vs hand-rules baseline | not measured | **+52 pp F1**, +96.8 pp recall |
| Content classifier F1 | 0.971 | 0.971 (unchanged — per-type audit confirms no hidden hotspot) |
| Cross-source PR-AUC drop (Windows) | not measured | **−21 pp** (footnote: within-source-inflated) |
| Cross-source PR-AUC drop (Linux) | not measured | −4 pp |
| Content classifier F1 on CredData (external) | not measured | **0.564** vs ShareSift-own-split 0.971 |

The honest story is that v0.5's *ranking* claims held everywhere, its
*ordering* of Snaffler held everywhere, but one of its three
operating-point claims (recall@0.5 vs Snaffler) was measured on a
non-deployed model. Once recalibrated against the OOD distribution,
the calibrated-deployed model actually beats Snaffler by more than the
original misreported figure — the audit *improved* the deployed claim
while making the methodology defensible.

## Why this is a methodological win

The 2026-05-30 audit caught data leakage and turned a fragile "beats
Wiz by 3-5 pp" into a defensible "beats Wiz by 13 pp on a leak-free
test set". This pass caught **metric mis-attribution + OOD calibration
collapse** and turned a fragile "+13.7 pp vs Snaffler" (on the wrong
model) into a defensible "+29.3 pp [CI +23.5 to +34.8]" (on the
deployed model with a proper paired bootstrap and a refit calibrator).

The pattern is the same: every numeric claim survives a hostile read
or gets footnoted, and the footnotes are themselves shipped. The cost
is one audit doc + six measurement tools + one model fix; the benefit
is that nobody using ShareSift in a serious context has to take a single
unverified claim on faith.
