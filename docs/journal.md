- First live run of `source_github` surfaced two real bugs in the un-unit-tested HTTP layer — exactly what manual smoke is for. Both fixed together (single commit) with the rate-limit state machine pulled into unit-testable shape, leaving only the actual `urlopen` call as manual-smoke. **Bug 1: pacing didn't respect the per-minute budget across pagination.** The old fixed-6.5s-spacing pacer only tracked the most recent single request via `_last_api_call_time`, so a multi-page query fired several requests in quick succession and blew the 10-per-minute cap within the first couple queries — the pacer's "9 req/min steady-state" math assumed one request per query rather than N pages × N queries against the same budget. Replaced with a **sliding-window deque** (`_REQUEST_TIMESTAMPS`) that tracks every request timestamp regardless of which query it belongs to; before each call, evicts entries older than 60s and waits for the oldest in-window entry to age out before exceeding `_BUDGET_PER_WINDOW` (9 per 60s, one under the hard cap for clock-skew safety). Plus a new `_honor_remaining_header` post-success hook that reads `X-RateLimit-Remaining` / `X-RateLimit-Reset` from every successful response and proactively sleeps until Reset + 5s buffer if Remaining ≤ 1 — the server's view of the budget is authoritative, which is what makes 9-per-60s safe vs needing the more conservative 8.
- **Bug 2: backoff silently dropped queries to zero results on exhaustion.** The old 1-rate-limit-retry budget was too tight against the actual reset window — when the retry also hit the limit (which happens when our wait was under-estimated), the previous code raised `RuntimeError` that propagated up and crashed the whole run OR returned a sparse partial response that terminated pagination early. Vincent observed the latter on his live run: query "silently skipped producing zero results." This was the load-bearing defect — never-silently-skip is the load-bearing fix. Three changes: (a) retry budget bumped to `_MAX_RATE_LIMIT_RETRIES = 3` (was 1) per page, so transient under-waits don't kill a query; (b) `_rate_limit_sleep_seconds` now uses `X-RateLimit-Reset` as the primary signal (was Retry-After; the latter is now fallback for the abuse-rate-limits we shouldn't hit at this volume), with a 5-second buffer (was 2) to absorb clock skew; (c) **per-query graceful failure** — `_search_one_query` catches `RuntimeError` from `_fetch_search_page`, updates the manifest with `status="failed_after_retries"` + error message naming the page, and raises a new `_QueryFailedError`; `collect()` catches that ONLY at the per-query boundary and records it in the new `CollectResult.failed_queries` list, continuing to the next query rather than crashing the run. ## 2026-05-31 (continued — v0.5 Tier-2.3: CredData benchmark landed, exposed content-classifier overfit)

Followed Tier 2.1 with the CredData run, since the download cost turned out to be ~6 minutes (4 parallel jobs) instead of the projected 30-60. Result is the single most consequential finding of the entire v0.5 audit pass — and it's a downward correction.

- **CredData download + obfuscation.** `cd data/external/creddata && uv run --with-requirements requirements.txt python download_data.py --jobs 4 --data_dir data`. Downloaded 333 external repos, obfuscation pass replaces real credentials with shape-preserving synthetic ones. 1.0 GB final corpus at `data/external/creddata/data/<repo-hash>/...` (excluded from git via the new gitignore class).

- **Benchmark construction (`tools/build_creddata_benchmark.py`).** Rewrote the framework to read from the local clone instead of `raw.githubusercontent.com`. Stratified sample 500 positives + 1000 negatives = 1500 records from the 66,896-row metadata aggregate. Snippets cap at 4000 chars (p99 was 21K chars, max 173K — single-line minified JS/HTML files); when over cap, truncate context lines symmetrically around LineStart..LineEnd, preserving the secret-bearing line. Output at `data/eval/creddata_benchmark.jsonl` in the same chat-template shape `tools/eval_content_classifier.py` consumes.

- **First eval attempt — OOM.** Without the snippet cap, hit `torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 1.38 GiB` at record ~50. Co-tenant on the GPU (qwen_cyber's llama-server at 23GB + hybrid_query.py at 2GB) left only 6GB headroom; the 173K-char snippet's KV cache exceeded that. Fix: snippet cap at build time + `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` for fragmentation safety.

- **Second eval — 1500 records, 22 min wallclock.** v0p3 on CredData: **TP=384, FP=478, FN=116, TN=522, P=0.445, R=0.768, F1=0.564, accuracy=0.604**. Prediction distribution skewed positive (862 yes / 638 no on a corpus with 500 true positives) — model overpredicts secrets by ~70%. Biringa & Kul 2025's Mistral-7B-Instruct-v0.3 LoRA on full CredData gets F1=0.985; our F1=0.564 is a 42-point gap.

- **Bug found and fixed.** `tools/eval_content_classifier.py:252` called `args.results_out.relative_to(REPO_ROOT)` where args.results_out was a relative path — raised ValueError after the JSON write succeeded but before the predictions JSONL write. Patched to `try/except ValueError` around the print so a relative path doesn't blow up the finalization. Results JSON has the row; predictions JSONL was lost on this run (re-runnable in another 22 min if needed).

- **Diagnosis.** Tier 1.D already flagged that ~96% of training positives are LLM-rule-labeled (regex matches on `password=`, `apikey=`, etc.), not kingfisher-validated. On ShareSift's own test split (drawn from the same labeling distribution), that doesn't matter and the F1=0.971 holds. On CredData, where the negative class is hand-curated test fixtures, parameter names, and documentation that *deliberately* look credential-shaped without being credentials, the model can't distinguish "mentions credentials" from "contains credentials" and FP=478. Recall stays moderately strong (0.77) because the model does recognize true positives, but precision collapses (0.45). This is the strongest argument so far for prioritizing the Tier 2.2 verified-positive corpus rebuild — the headline F1 isn't portable to externally-curated benchmarks.

- **Honest writeup.** Updated `docs/audit_2026-05-31.md` to fold the CredData result into the headline-shifts table and to add a dedicated §"Tier-2.3 finding" section with the gap, the mechanism, the caveats (model-size asymmetry, sample-vs-full, training distribution mismatch), and the v0.6 target (close the gap to ≥0.85). README updated to footnote the F1=0.971 claim with the CredData F1=0.564 counterpoint.

- **v0.5 final status after Tier 2.3.** Path classifier (Windows + Linux): honest end-to-end, well-calibrated, significantly beats Snaffler on the production operating point, generalizes within-source. Content classifier: strong on ShareSift's own distribution, doesn't generalize to externally-curated benchmarks — flagged as the v0.6 priority. v0.5 ships as "credible foundation with one well-quantified known limitation" rather than "fully validated on field-comparable benchmarks".
